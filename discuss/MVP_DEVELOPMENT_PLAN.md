# 智答课堂 AI MVP 开发计划

## 1. 开发原则

- 一次只开发一个可验收模块。
- 每个模块都必须有明确验收标准。
- 先跑通业务闭环，再优化体验和性能。
- 先用扫码模拟器打通课堂流程，再接入真实移动端扫码。
- 所有运行和调试动作通过 `scripts/` 下的 `.sh` 脚本执行。
- 文档、接口、数据模型、测试同步推进。

## 2. 阶段拆分

### 阶段 0：工程骨架

交付：

- 项目目录结构。
- 文档目录。
- 讨论目录。
- 脚本目录。
- 日志目录。
- 结构检查脚本。

验收：

- 根目录保持简洁。
- `docs/` 存放正式文档。
- `discuss/` 存放方案和评审材料。
- `scripts/` 存放 `.sh` 脚本。

### 阶段 1：后端基础

交付：

- NestJS API 服务。
- Prisma 数据模型。
- PostgreSQL 连接配置。
- 用户、班级、学生、卡片、题目、课堂活动、答题记录模块。
- 统一日志输出到 `logs/`。

验收：

- 可以通过脚本启动 API。
- 可以通过脚本运行测试。
- API 返回结构稳定。
- 关键接口有 DTO 校验。
- 不在 Controller 中写业务逻辑。

### 阶段 2：教师 Web 端

交付：

- 登录页。
- 班级管理。
- 学生导入。
- 卡片绑定。
- 题库管理。
- 课堂活动创建。
- 报告查看。

验收：

- 教师能完成一次出口检测的准备工作。
- 表单有必要校验。
- 页面不出现学生隐私泄露。

当前进展：

- 已新增班级管理页，支持班级增删改查。
- 已新增学生批量导入、学生编辑和删除。
- 已新增题库管理页，支持题目增删改查。
- 已新增题目批量导入。
- 教师可在管理页选用班级和题目后开始课堂活动。

### 阶段 3：课堂大屏端

交付：

- 活动展示页。
- 题目展示。
- 答题进度。
- 匿名选项统计。
- 答案解析。
- AI 讲评建议展示。

验收：

- 大屏不展示学生姓名。
- 扫码结果提交后统计能实时更新。
- 字体和布局适合投影展示。

当前进展：

- 已新增 `apps/web-screen` 大屏端。
- 已支持无参数演示数据展示，便于独立投影验收。
- 已支持通过 `sessionId`、`questionId` URL 参数读取真实课堂活动、题目统计和课堂报告。
- 已新增 `scripts/web/screen-dev.sh`、`scripts/web/screen-build.sh`。
- 教师 Web 端创建活动后会生成课堂大屏地址。
- 大屏已支持每 3 秒自动刷新统计、报告和学生答题状态。
- 大屏已支持展示已答/未答学生名单，默认使用脱敏后的 `displayName`。
- 教师 Web 端已新增“大屏显示原始姓名”开关，打开后大屏地址携带 `showRealNames=true`。

### 阶段 4：扫码模拟器

交付：

- 开发用扫码模拟页面或脚本。
- 批量提交答题结果。
- 重复卡号处理。
- 识别置信度字段。

验收：

- 60 名学生答案可以批量提交。
- 同一卡号重复提交时以最新答案为准。
- 大屏统计正确更新。

当前进展：

- 已新增 `apps/scan-simulator` 扫码模拟器。
- 已支持一键创建 60 人班级、题目和进行中的出口检测活动。
- 已支持批量提交 60 张答题卡。
- 已支持重复提交同一卡号，验证后端按最新答案覆盖。
- 已新增 `scripts/simulator/dev.sh`、`scripts/simulator/build.sh`。

### 阶段 5：AI MVP

交付：

- AI 生成题目。
- AI 错因分析。
- AI 讲评建议。
- AI 生成记录。
- AI 输出结构校验。

验收：

- AI 输出合法 JSON。
- AI 题目必须进入待审核状态。
- AI 分析使用匿名或聚合数据。
- 页面显示“AI 生成，教师审核后使用”。

当前进展：

- 已支持 AI 题目生成并记录生成行为。
- 已新增 `POST /api/ai/sessions/{sessionId}/diagnose`，基于匿名聚合统计生成结构化错因诊断。
- 已新增 `GET /api/ai/records` 查询 AI 生成记录。
- 大模型未配置时会返回规则兜底诊断，保证开发环境可验收。
- 教师端和大屏端已展示错因、依据、讲评建议和后续动作。

### 阶段 6：真实扫码原型

产品决策：

- 正式课堂扫码使用教师手机 App。
- 教师 Web 端负责备课、活动创建和报告查看。
- 课堂大屏 Web 负责投影展示。
- 网页摄像头只作为开发期扫码调试工具，不作为正式课堂扫码入口。

交付：

- Flutter 教师端 App。
- 摄像头调用。
- `ZhiDaCard v1` 纸卡识别。
- 方向转换 A/B/C/D。
- 多帧确认。
- 离线缓存。
- 批量上传。

验收：

- 可识别 40 张测试卡。
- 能处理重复识别。
- 离线缓存可以恢复上传。

当前进展：

- 已完成纸质答题卡版式规范 `docs/CARD_LAYOUT_SPEC.md`。
- 已定义 `ZhiDaCard v1` 黑白方形码：7×7 网格、10 位卡号、4 位校验、7 位版本位。
- 已生成 60 张可打印测试卡 `docs/card-layout/zhida-cards-v1.html`。
- 已新增 `scripts/cards/generate.sh` 生成测试卡。
- 已新增 `scripts/cards/verify.sh` 校验 `C001-C060` 编码与解码一致。
- 已新增 `services/cv-service` 的 `ZhiDaCard v1` 解码核心，支持卡号、A/B/C/D 方向和置信度输出。
- 已新增多帧确认逻辑，连续多帧一致后才生成可上传答案。
- 已新增灰度图采样逻辑，支持从已定位四角的码图采样 7×7 网格。
- 已新增 MVP 级码定位器和 `ScanFramePipeline`，可从灰度图完成定位、采样、解码和多帧确认。
- 已新增 `AnswerBatchCollector`，可把确认答案聚合为后端批量答题请求体，并处理同一卡号覆盖。
- 已新增 `ScanSession`，将连续视频帧转换为后端批量上传 payload。
- 已新增 `apps/scan-debug` 扫码调试台，用于在 Flutter 接入前可视化验证卡号、方向、多帧确认和上传 payload。
- 扫码调试台已支持粘贴真实 `sessionId`、`questionId` 后上传到后端批量答题接口。
- 教师 Web 端创建活动后会生成扫码调试台地址，并自动携带 `sessionId`、`questionId` 和 `deviceId` 参数。

## 3. 下一步任务

优先进入阶段 1：

1. 初始化后端工程。
2. 建立数据库模型。
3. 编写 `scripts/api/dev.sh`、`scripts/api/test.sh`。
4. 实现班级、学生、卡片、题目、课堂活动最小接口。
5. 使用测试验证出口检测核心数据流。
