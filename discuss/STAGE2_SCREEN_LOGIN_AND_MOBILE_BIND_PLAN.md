# 第二阶段：大屏登录与手机免登录绑定方案

## 目标

第二阶段的边界是：大屏端必须先登录用户系统，手机端不登录，但必须通过大屏二维码中的 `sessionId + bindToken` 进入指定课堂。

手机端拿到绑定信息后，只作为当前课堂的扫码控制端使用，不能进入后台管理入口。

## 绑定模型

大屏端拿到当前课堂后，向后端请求课堂绑定信息。后端返回：

- `sessionId`：课堂 ID
- `bindToken`：后端签名生成的课堂绑定令牌
- `mobileBindUrl`：教师手机端 URL，包含 `sessionId` 和 `bindToken`

手机端保存这两个参数，并在后续请求里携带：

- `X-Zhida-Mobile-Session-Id`
- `X-Zhida-Mobile-Bind-Token`

后端校验令牌后，把请求临时映射到该课堂所属的 `workspaceId` 和 `teacherUserId`，从而复用现有的数据隔离逻辑。

## 允许的手机能力

第一版先保留课堂内需要的能力：

- 读取当前课堂实时状态
- 上传扫码答案
- 切换当前题与评测轮次
- 查看本轮与整堂报告
- 在当前课堂上下文里选题、新增题目和 AI 生成题目

手机端不展示班级管理、学生管理、题库管理等后台入口。

## 兼容策略

旧的 `displayPairCode` 配对流程暂时保留，避免中断当前真机测试。新流程中，只要大屏已绑定课堂，展示给手机的二维码将使用 `sessionId + bindToken`。

后续可以在确认新流程稳定后，逐步移除手机端创建课堂的旧入口。
